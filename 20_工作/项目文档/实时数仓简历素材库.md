# 实时数仓简历素材库

> 从两篇技术文档中提炼的所有可用于简历的素材点，按能力维度分类。每个点标注了简历价值评级（S/A/B/C）和适用场景。

## 一、架构设计能力

这是最能体现高级别工程师水平的维度，面试官最关注"你为什么这么设计"。

### 1.1 三引擎混合架构设计（S 级）

**素材：** 设计了 Flink 流引擎 + Doris 批引擎 + Blade 状态引擎的混合架构，并制定了基于时效性、数据量级、模型复杂度、状态大小四个维度的引擎选型决策树。

**理由：** 这是整个项目最大的架构亮点。绝大多数团队只用 Flink 或者 Flink+Doris，三引擎混合且有清晰选型标准的非常少见。体现了你对不同计算引擎优劣势的深刻理解，以及面对复杂场景时的架构权衡能力。

**面试追问方向：** 三种引擎各自的适用场景？Blade 引擎解决了什么问题？为什么不用 Cellar/Redis 替代 Blade？

### 1.2 流批切换点设计（A 级）

**素材：** 文档中明确提出了四种"流转批"切换点——从源数据层切换（适合 10 分钟级时效、小数据量、极复杂 ETL）、从事实层切换（5 分钟级、中等数据量）、从轻度汇总层切换（分钟级、较大数据量）、从应用层切换（对外提供 OLAP 服务）。

**理由：** 这个点非常有区分度。大多数人只知道"流批一体"的概念，但很少有人能说清楚在数仓的哪一层做切换、为什么在这一层切。这体现了你对数据流转全链路的掌控力。

**面试追问方向：** 怎么判断在哪一层切换？切换后数据一致性怎么保证？

### 1.3 实时 OLAP 变体架构（A 级）

**素材：** 在服务层采用"实时 OLAP 变体"，将部分聚合、分析、计算下推到 Doris OLAP 引擎承担，使主题层和应用层计算更轻量，从而灵活应对业务分析需求的频繁变更。

**理由：** 这是一个很聪明的架构决策。传统做法是在 Flink 里算好所有指标再写入存储，但业务需求变化快时每次改指标都要改 Flink 任务、重新上线。把部分计算下推到 Doris，利用 OLAP 引擎的灵活查询能力，需求变更时只改 SQL 查询即可，不用动实时任务。

**面试追问方向：** 哪些计算适合下推到 OLAP？下推后性能怎么保证？和预计算方案的 trade-off？

### 1.4 双链路跨机房容灾架构（A 级）

**素材：** 核心任务全链路（Binlog 采集 → Kafka → Yarn → Doris）跨机房主备部署（云谷/兆丰），主备链路保持配置一致、代码版本一致，支持热切换。

**理由：** 实时数仓做到跨机房容灾的团队不多，这体现了对高可用架构的理解。关键亮点是"全链路无单点"——不是只在某一层做备份，而是从数据采集到存储每一层都有独立的备份链路。

**面试追问方向：** 主备数据一致性怎么保证？切换时有没有数据丢失？双链路的资源成本怎么控制？

## 二、数仓建模能力

数据开发岗的核心硬技能，面试必问。

### 2.1 批流一体化建模体系（S 级）

**素材：** 实时与离线统一主题域划分、统一模型分层（ODS → Fact → Topic → App）、统一命名规范。Fact 层字段定义和口径完全一致，差异只在计算引擎和存储介质。

**理由：** "批流一体"是行业热词，但真正落地的团队很少。你能说清楚怎么做到统一（三个统一），以及这样做的好处（模型复用、实离校验简化），就比大多数候选人强。

**面试追问方向：** 实时和离线的模型完全一样吗？有哪些不得不不一样的地方？怎么处理的？

### 2.2 四层模型分层设计（A 级）

**素材：** ODS（原始数据接入）→ Fact（清洗过滤关联扩维，星型模型）→ Topic（跨域关联聚合，主题宽表）→ App（面向分析主题，对外服务）+ Dim（离线初始化 + 实时写入，Redis/Tair 存储）。

**理由：** 相比常见的三层（ODS/DWD/ADS），这里多了一个 Topic 主题汇总层，专门做跨域关联和聚合。这个设计决策值得讲——为什么要多加一层？因为物流业务跨多主题分析场景多，如果没有 Topic 层，每个 App 层任务都要自己做跨域关联，重复开发且不一致。

**面试追问方向：** Topic 层和 App 层的边界怎么划分？什么时候该沉淀到 Topic 层？

### 2.3 维度表实时化方案（B 级）

**素材：** 维度层采用"离线数据初始化 + 实时数据写入"的方式，使用 Redis/Tair 作为维度存储。

**理由：** 实时数仓的维度表处理是个经典难题。离线维度表更新不及时，纯实时维度表又可能丢历史数据。这个"离线打底 + 实时增量"的方案是业界最佳实践，体现你对这个问题的理解。

**面试追问方向：** 维度表更新延迟怎么处理？缓存一致性怎么保证？维度表太大 Redis 放不下怎么办？

### 2.4 建模思想体系（B 级）

**素材：** 文档总结了七条建模思想——层次精简化、模型宽表化、高内聚低耦合、公共逻辑下沉、单一职责、命名规范清晰、指标统一管理、分离易错点。

**理由：** 这些思想本身不算新，但如果你能结合实际案例讲清楚每一条是怎么落地的（比如"公共逻辑下沉"具体下沉了什么、解决了什么问题），就能体现你不是纸上谈兵。

## 三、Flink 技术深度

面试官最喜欢追问的技术细节，能体现你是"用过"还是"真懂"。

### 3.1 State 管理与 OOM 治理（S 级）

**素材：** 针对 RocksDB State 堆外内存不受 JVM 管控导致 Container 被 kill 的问题，通过 Flink 1.12 升级（更严格的内存控制）+ JVM Overhead 调大（25%→30%-40%）+ 持久化 Checkpoint 替代 Savepoint（减少磁盘压力）组合解决。

**理由：** 这是 Flink 生产环境中最常见也最棘手的问题之一。能把根因（堆外内存不受 JVM 管控）、排查过程、解决方案讲清楚，直接证明你有大规模 Flink 生产运维经验。

**面试追问方向：** RocksDB 的内存模型是什么？Checkpoint 和 Savepoint 的区别？增量 Checkpoint 的原理？

### 3.2 Join 算子深度优化（S 级）

**素材：** 深入到 Flink Join 算子的三种实现——JoinKeyContainsUniqueKey（ValueState，最优）、InputSideHasUniqueKey（MapState，Key 是主键）、InputSideHasNoUniqueKey（MapState，Key 是整行数据，最差）。生产中优先保证关联字段是主键，最大化优化 State 读写效率。

**理由：** 这个深度非常少见。绝大多数 Flink 开发者只知道 Regular Join / Interval Join / Window Join 的区别，但不知道 Regular Join 内部还有三种不同的 State 存储策略。能讲到这个层面，直接拉开和其他候选人的差距。

**面试追问方向：** 三种 Join 的 State 大小差异有多大？怎么判断当前任务用的是哪种？如何改造 SQL 让它走最优路径？

### 3.3 Deduplication 替代 ROW_NUMBER 去重（A 级）

**素材：** 文档提到使用 Deduplication 算子替代 ROW_NUMBER 进行去重，性能更优。

**理由：** 这是 Flink SQL 调优的一个实用技巧。ROW_NUMBER 去重在底层会生成 TopN 算子，而 Deduplication 是专门为去重优化的算子，State 更小、性能更好。能说出这个区别，体现你对 Flink SQL 执行计划有研究。

**面试追问方向：** 两者在执行计划上的区别？Deduplication 的触发条件是什么？

### 3.4 多流关联方案选型（A 级）

**素材：** 对比了 Window Join、Interval Join、Regular Join 三种方案，结合物流业务特点（关联周期长 2-3 天、不允许丢数），选择了 Regular Join。

**理由：** 多流关联是实时数仓的核心难题。能说清楚三种 Join 的适用场景和 trade-off（Window Join 丢数据、Interval Join 有时间窗口限制、Regular Join State 大但不丢数），体现你的技术选型能力。

### 3.5 性能压测方法论（B 级）

**素材：** 上线前做流量回放压测，目标一是确保流量高峰（单表 QPS 4W+，持续 5 分钟的"集单"场景）不出现 Checkpoint 连续失败和长时间反压；目标二是量化任务处理能力（某并行度下单位时间最大处理量），为容量规划提供依据。

**理由：** 大多数团队的实时任务是"上线了再说"，有系统化压测机制的不多。这体现了工程化思维。

### 3.6 Blade 引擎状态外置方案（B 级）

**素材：** 将 Flink 的中间计算状态外置到 Blade 数据库，利用 Blade 的水平扩展能力解决千万级大状态问题。多流关联时每条流只更新 Blade 表的部分字段，不再在 Flink 内部保存状态。再通过 Blade Binlog 收集将存储数据重新转为流。

**理由：** 这是一个非常有创意的方案——把"有状态计算"变成"无状态计算 + 外部存储"。虽然 Blade 是内部组件，但这个思路可以迁移到任何支持 CDC 的数据库（如 MySQL + Canal、TiDB + TiCDC），面试时可以抽象来讲。

## 四、数据质量治理能力

越来越多的公司把数据质量作为数据团队的核心 KPI，这个维度的素材很有价值。

### 4.1 全生命周期质量保障体系（S 级）

**素材：** 建立了"故障预防 → 故障发现 → 故障恢复 → 故障改进"四环节闭环体系。

**理由：** 这是一个完整的方法论，不是零散的优化点。面试时能画出这个框架，再往每个环节里填具体措施，非常有说服力。体现你不只是"救火"，而是有体系化的治理思维。

### 4.2 三层数据监控体系（A 级）

**素材：** 多源校验（不同分层间数据对比 + 主备对比）、实离校验（实时与离线同口径对比）、波动校验（同比环比 + 断流监控）。

**理由：** 三层监控覆盖了数据质量的三个核心维度——一致性（多源校验）、准确性（实离校验）、及时性（波动校验）。这个分类方式本身就很有价值，体现你对数据质量问题的系统性思考。

**面试追问方向：** 实离校验的延迟怎么处理（实时和离线产出时间不同）？波动校验的阈值怎么设定？误报怎么处理？

### 4.3 任务分级保障机制（A 级）

**素材：** 将任务分为 S 级和非 S 级，制定差异化 SLA：S 级任务故障发现 3 分钟、恢复 6 分钟、故障频率 < 1 次/60 天，配备电话报警 + 双链路 + 生产链路监控 + 平台 SLA + 高优集群；非 S 级任务故障发现 10 分钟、恢复 20 分钟、故障频率 < 1 次/45 天，大象/短信报警、普通集群。

**理由：** 资源永远是有限的，不可能所有任务都做最高级别保障。分级保障体现了你的资源分配和优先级判断能力。而且这里有非常具体的 SLA 数字，面试时直接说出来非常有说服力。

### 4.4 开发流程规范化（B 级）

**素材：** 文档指出早期"coding 时长占整体开发时长过高，技术方案评审、code review、数据验证、运维体系搭建的时间被压缩"。后期建立了完整的开发流程：技术方案评审 → 开发 → Code Review → 性能压测 → 数据验证 → 监控配置 → 上线。

**理由：** 这个点适合在面试中讲"你推动了什么流程改进"。从"野蛮生长"到"规范化开发"，体现你的工程管理意识。

### 4.5 实时 DQC 建设（B 级）

**素材：** 建设了实时数据质量检查（DQC）体系，包括事前的开发规范检查和测试环境建设，事后的实时 DQC 监控。

**理由：** DQC 是数据治理的标配能力，但实时 DQC 比离线 DQC 难度大很多（数据是流式的，不能像离线那样跑完再检查）。如果你能讲清楚实时 DQC 的实现方式，是加分项。

## 五、集群运维与性能调优

体现你不只是写代码，还能管好线上环境。

### 5.1 Yarn 集群调优（A 级）

**素材：** 三个关键措施——控制集群资源使用率 80% 以内、多磁盘部署分散 State IO 压力、持久化 Checkpoint 替代 Savepoint 降低磁盘写入。

**理由：** 这些是实打实的生产经验，不是看文档能学到的。尤其是"多磁盘部署"这个点，说明你理解 Flink State 的 IO 模型（RocksDB 是 LSM-Tree，写放大严重，对磁盘 IO 要求高）。

### 5.2 Kafka 集群治理（B 级）

**素材：** 集群物理隔离（物流业务独立 Kafka 集群）、定时扫描 topic partition 数据量避免倾斜。

**理由：** Kafka 集群隔离是保障数据可靠性的基础措施。partition 倾斜扫描体现了你对 Kafka 运维的关注。

### 5.3 Doris 集群治理（A 级）

**素材：** 四个措施——集群隔离（不同业务独立集群）、接口层禁止大查询 + 使用缓存、Doris2Doris 任务错峰执行（时间分桶）、限制 Doris 开放（生产集群只允许数据 RD 查询）。治理后 TP95 从 2500ms 降至 1300ms，磁盘 IO 从 70% 降至 30%。

**理由：** Doris 是近年来数仓领域的热门技术，很多公司在用。你能讲出 Doris 的性能瓶颈在哪（事务数过大、大查询拖慢集群）以及怎么治理，说明你有 Doris 的深度使用经验。

### 5.4 Doris2Doris 批处理的性能边界（B 级）

**素材：** 文档给出了具体的性能数据——百万级数据量 ETL 约 1 分钟，五千万级约 5 分钟。同时指出了批引擎的三个瓶颈：分钟级延时、大数据量读写性能瓶颈、集群计算能力短板。

**理由：** 这些是实际生产中的经验数据，面试时能说出具体的性能边界（而不是"大概几分钟"），体现你对技术的量化认知。

## 六、开发效率提升

体现你不只是完成需求，还在思考怎么让团队更高效。

### 6.1 FlinkSQL 零 Java 代码开发（A 级）

**素材：** 拥抱 NAU 实时开发平台，实现实时开发零 Java 代码，全部使用 FlinkSQL。同时总结了两条代码复用规律，将通用逻辑模板化。

**理由：** 从 Java DataStream API 到 FlinkSQL 的迁移不是简单的"换个写法"，需要解决很多 SQL 表达不了的场景。你能推动团队完成这个迁移，说明你对 FlinkSQL 的能力边界有清晰认知。

### 6.2 代码复用与模板化开发（B 级）

**素材：** 发现两条复用规律——大部分需求是新指标逻辑的再生产（可模板化）、部分通用逻辑可以抽象为公共组件。基于此建立了模板化开发体系。

**理由：** 这体现了"工程化思维"——不是每次都从零开始写，而是把重复的部分抽象出来。面试时可以讲具体的模板设计和复用率数据。

### 6.3 统一数据服务平台（B 级）

**素材：** 针对服务层口径管理难、取数模块与数仓耦合不清的问题，搭建了统一数据服务平台，统一管理指标口径和数据服务。

**理由：** 这个点适合讲"你发现了什么问题、怎么推动解决的"。指标口径不一致是数据团队的经典痛点，能推动建设统一平台说明你有全局视野。

## 七、业务理解能力

数据开发不是纯技术岗，业务理解是加分项。

### 7.1 物流全链路业务理解（A 级）

**素材：** 文档详细描述了优选物流的完整链路——共享仓 → 中心仓 → 网格站 → 团长，以及"预购+自提"的业务模式、"集单"的业务动作、跨天夜间作业的特殊性。

**理由：** 面试时能用 2-3 句话讲清楚业务背景，说明你不是"闷头写代码"的人。尤其是"集单"场景（固定时间点的流量洪峰）和"跨天作业"（导致离线只能看 T-2 数据）这两个点，直接解释了为什么需要实时数仓，非常有说服力。

### 7.2 数据直接参与生产决策（A 级）

**素材：** 实时数据不只是用于 BI 分析，而是直接参与仓储收发货调度、运输排线优化、供应商结算（涉及资金）等生产环节。

**理由：** 这大幅提升了项目的含金量。"做个看板给老板看"和"数据错了供应商就多结/少结钱"完全是两个难度级别。面试时一定要强调这一点。

## 素材使用建议

### 按岗位级别选择素材

**初级/中级数据开发（P5-P6）：** 重点用第三部分（Flink 技术深度）和第二部分（数仓建模），证明你的技术硬实力。选 2-3 个 S/A 级素材深入展开。

**高级数据开发（P6-P7）：** 在技术深度基础上，加入第一部分（架构设计）和第四部分（数据质量治理），体现你的架构能力和体系化思维。
